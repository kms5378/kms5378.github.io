## 오늘 다룬 내용

오늘은 운영체제가 왜 유저 프로그램에게 물리주소를 직접 쓰게 하지 않는지, 그리고 PintOS에서 유저 프로그램을 실행할 때 만나는 ELF가 무엇인지 정리했다.

처음 질문은 메모리 보호에 대한 것이었다. 프로세스 A가 물리주소를 직접 알고 있다면 프로세스 B의 메모리나 커널 메모리도 건드릴 수 있다. 그렇다면 사용 중인 메모리에 lock을 걸어서 막을 수 있지 않을까 하는 의문이 있었다.

결론부터 말하면 lock으로는 막을 수 없다. lock은 여러 코드가 같은 규칙을 지킨다는 전제에서 동작하는 소프트웨어 약속이다. 반면 메모리 보호는 악의적인 코드나 버그 있는 코드도 강제로 막아야 한다. 그래서 운영체제는 lock이 아니라 CPU의 유저 모드, MMU, 페이지 테이블 같은 하드웨어 보호 장치를 사용한다.

## 핵심 개념

lock은 동시성 문제를 줄이기 위한 도구다. 예를 들어 커널 내부에서 공유 자료구조를 수정할 때 `lock_acquire()`와 `lock_release()`로 critical section을 감쌀 수 있다.

하지만 유저 프로그램이 물리주소에 직접 load/store 할 수 있다면 이야기가 달라진다.

```c
*(int *)0x00100000 = 123;
```

이 코드가 실제 물리주소 `0x00100000`에 바로 쓰는 명령이라면, OS가 중간에서 검사할 기회가 없다. 메모리 접근은 CPU가 실행하는 명령이고, lock을 잡았는지 확인하는 별도 절차를 자동으로 거치지 않는다.

심지어 물리주소 접근이 가능하면 lock 자료구조 자체도 메모리에 있으므로, lock을 무시하거나 lock 값을 깨뜨릴 수도 있다. 따라서 lock은 "규칙을 지키는 코드끼리 순서를 맞추는 장치"이지, "허가받지 않은 주소 접근을 막는 보안 장치"가 아니다.

실제 보호는 가상주소를 통해 이루어진다.

```text
process A virtual 0x8048000 -> physical 0x12345000
process B virtual 0x8048000 -> physical 0x12346000
kernel    virtual 0xC0100000 -> physical 0x00100000
```

유저 프로그램은 물리주소를 직접 사용하는 것이 아니라 가상주소를 사용한다. CPU는 메모리에 접근할 때마다 현재 프로세스의 페이지 테이블을 보고, 그 가상주소가 실제 물리 페이지에 매핑되어 있는지 확인한다. 이때 유저 모드에서 접근 가능한 페이지인지, 쓰기 권한이 있는지, 커널 전용 페이지는 아닌지도 함께 검사한다.

조건을 만족하지 못하면 CPU는 page fault를 발생시킨다. 이 지점이 중요하다. 보호는 유저 코드가 착하게 동작하기를 기대하는 것이 아니라, CPU가 매 접근마다 권한을 강제하는 방식으로 이루어진다.

## 물리주소를 직접 쓰면 왜 막을 수 없는가

오늘 가장 중요한 정리는 다음이었다.

```text
유저가 물리주소를 직접 접근 가능함
-> 소프트웨어만으로 보호 불가능

유저는 가상주소만 사용 가능함
-> MMU와 페이지 테이블로 보호 가능
```

운영체제는 "물리주소 접근을 허용한 뒤 나쁜 접근만 골라서 막는" 방식으로 보호하지 않는다. 애초에 유저 모드에서는 물리주소를 직접 접근하지 못하게 하고, 유저가 보는 주소를 전부 가상주소로 만든다.

커널은 privileged mode에서 페이지 테이블을 만들고 수정할 수 있다. 유저 프로세스는 자기 페이지 테이블에 매핑된 유저 영역만 접근할 수 있다. 다른 프로세스의 물리 페이지나 커널 페이지가 현재 유저 페이지 테이블에 없거나 유저 접근 불가로 표시되어 있으면, 해당 주소를 건드리는 순간 page fault가 난다.

OS에서 "메모리를 lock한다"는 표현이 나와도 의미를 구분해야 한다. 어떤 페이지를 swap out하지 못하게 pinning하거나, 커널 내부 자료구조 경쟁을 막기 위해 lock을 잡는 것은 가능하다. 하지만 그것은 다른 유저 프로세스가 물리주소를 직접 쓰지 못하게 만드는 보호 장치와는 다른 문제다.

## ELF 이해하기

이후에는 ELF가 무엇인지 정리했다. ELF는 Executable and Linkable Format의 약자다. 리눅스나 유닉스 계열에서 실행 파일, 오브젝트 파일, 공유 라이브러리 등에 쓰는 표준 파일 형식이다.

쉽게 말하면 ELF는 운영체제 로더가 읽는 "실행 파일의 포장 형식"이다. C 코드를 컴파일해 나온 실행 파일 안에는 단순히 기계어 코드만 들어 있는 것이 아니라, OS가 그 파일을 어떻게 메모리에 올리고 어디서부터 실행해야 하는지에 대한 정보도 들어 있다.

ELF에서 중요한 정보는 대략 다음과 같다.

- ELF header: ELF 파일인지 확인하는 magic number, 32비트/64비트 여부, entry point, program header 위치
- Program headers: 실행 시 어떤 segment를 어떤 가상주소에 올릴지, 읽기/쓰기/실행 권한은 무엇인지
- Sections: `.text`, `.data`, `.bss`, `.rodata`, `.symtab` 같은 컴파일/링크 관점의 구역

운영체제 로더 입장에서 특히 중요한 것은 program header다. 실행할 때 실제로 메모리에 올려야 하는 segment 정보를 담고 있기 때문이다.

예를 들어 ELF가 다음 정보를 가진다고 생각할 수 있다.

```text
.text -> virtual 0x08048000, read + execute
.data -> virtual 0x0804a000, read + write
.bss  -> virtual 0x0804b000, read + write, zero fill
entry -> virtual 0x08048320
```

그러면 로더는 파일에서 필요한 부분을 읽어 유저 가상 메모리에 올리고, 남은 공간은 0으로 채우며, 페이지 테이블을 설정한다. 마지막에는 CPU가 실행을 시작할 instruction pointer를 ELF의 entry point로 맞춘다.

PintOS에서 `load()` 함수가 하는 일이 바로 이 흐름과 연결된다. ELF header를 확인하고, `PT_LOAD` segment를 골라 유저 가상주소에 매핑하며, 최종적으로 유저 프로그램이 시작할 준비를 만든다.

## 헷갈렸던 점

처음에는 "사용 중인 메모리라면 lock을 걸어 접근을 막을 수 있지 않나"라고 생각했다. 하지만 lock은 접근 권한 자체를 CPU 수준에서 막지 않는다. lock은 이미 같은 주소를 합법적으로 접근할 수 있는 코드들이 순서를 맞추기 위한 장치다.

메모리 보호에서 중요한 기준은 "누가 lock을 잡았는가"가 아니라 "현재 CPU 모드와 페이지 테이블 권한상 이 가상주소 접근이 허용되는가"이다.

또 하나 정리한 점은 ELF를 단순한 실행 파일 이름 정도로 보면 부족하다는 것이다. ELF는 로더에게 다음 정보를 알려준다.

- 어떤 부분을 메모리에 올릴지
- 어느 가상주소에 올릴지
- 읽기, 쓰기, 실행 권한은 어떻게 줄지
- 어디서부터 실행을 시작할지

따라서 PintOS의 `load()`를 볼 때는 "파일을 읽는다"보다 "ELF 정보를 기준으로 유저 가상 메모리를 구성한다"라고 이해하는 편이 더 정확하다.

## 오늘의 결론

오늘의 핵심은 메모리 보호가 소프트웨어 약속이 아니라 하드웨어가 강제하는 권한 검사라는 점이다.

유저 프로세스가 물리주소를 직접 접근할 수 있다면 OS는 안전하게 보호할 수 없다. 그래서 유저 프로그램은 가상주소만 사용하고, CPU와 MMU는 페이지 테이블을 통해 매 접근마다 권한을 검사한다.

ELF는 이 보호 구조 위에서 실행 파일을 어디에 어떻게 올릴지 알려주는 형식이다. PintOS의 `load()`는 ELF를 읽어 유저 가상 메모리를 구성하고, 페이지 테이블과 entry point를 준비해 유저 프로그램 실행으로 이어준다.

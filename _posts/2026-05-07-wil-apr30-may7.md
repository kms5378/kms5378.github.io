## 이번 주에 한 일

이번 주는 학습 내용을 흘려보내지 않기 위한 기록 구조를 먼저 만들고, PintOS Project 2의 argument passing 흐름을 깊게 이해하는 데 집중했다.

GitHub Pages와 Jekyll 기반으로 TIL/WIL 블로그를 시작했다. `_posts` 폴더에 Markdown 파일을 두고, 파일 이름은 `YYYY-MM-DD-title.md` 형식으로 관리한다는 점을 정리했다. 카테고리는 TIL, WIL처럼 큰 흐름을 나누고, 태그는 PintOS, OS, Jekyll처럼 글의 핵심 키워드를 붙이는 방식으로 정했다. 이후 반복해서 사용할 WIL 회고 템플릿도 만들었다.

기술 학습 쪽에서는 PintOS Project 2의 `args-*` 테스트를 중심으로 argument passing을 다시 봤다. 처음에는 인자를 공백 기준으로 나누는 문제처럼 보였지만, 실제 핵심은 커맨드라인을 파싱한 뒤 유저 프로그램이 `argc`, `argv`를 제대로 받을 수 있도록 스택과 레지스터를 구성하는 일이었다.

특히 다음 실행 흐름을 하나로 연결해서 이해했다.

```text
process_create_initd()
-> initd()
-> process_exec()
-> load()
-> setup_stack()
-> do_iret()
-> user program _start(argc, argv)
```

`filesys_open()`에는 전체 커맨드라인이 아니라 실행 파일 이름인 첫 번째 토큰만 들어가야 한다. 하지만 유저 스택에는 실행 파일 이름을 포함한 전체 토큰 목록이 `argv` 형태로 들어가야 한다. 즉 파일을 여는 값과 유저 프로그램에 전달할 값이 서로 다르다는 점이 이번 주의 중요한 기준점이었다.

## 배운 점

argument passing은 단순한 문자열 파싱 문제가 아니었다. `strtok_r`로 토큰을 나누고, `argc`와 `argv`를 구성하고, 그 값을 유저 스택에 올리고, 마지막으로 `%rdi`, `%rsi` 레지스터에 `argc`, `argv`를 넘기는 흐름이 끝까지 맞아야 한다.

`strtok_r`는 원본 문자열을 직접 수정한다. `"echo hello world"`를 파싱하면 내부적으로 `"echo\0hello\0world\0"`처럼 바뀐다. 그래서 같은 커맨드라인을 실행 파일 이름 확인과 인자 목록 구성에 모두 사용하려면, 복사본을 어떻게 둘지 먼저 정해야 한다.

또 `strtok_r`가 토큰 배열을 자동으로 만들어주는 것이 아니라는 점도 정리했다. 반환값은 이번에 찾은 토큰의 시작 주소일 뿐이고, `argv[0]`, `argv[1]` 같은 포인터 배열은 직접 만들어야 한다. 그래서 `argv`는 문자열 배열이라기보다 문자열의 시작 주소들을 담는 배열로 이해하는 편이 더 정확했다.

스택을 구성할 때는 스택이 낮은 주소 방향으로 자란다는 점이 중요했다. 최종 메모리에서 `argv[0]`, `argv[1]`, `NULL` 순서로 보이게 하려면 포인터들은 뒤에서부터 push해야 한다. 문자열의 글자를 거꾸로 넣는 것이 아니라, 문자열 주소들을 역순으로 넣어야 최종 배열 순서가 맞는다.

x86-64에서는 함수 인자가 레지스터로 전달되므로 `_start(argc, argv)`에 맞춰 `argc`는 `%rdi`, `argv` 시작 주소는 `%rsi`로 들어가야 한다. 그리고 테스트가 출력만 보는 것이 아니라 스택 정렬까지 민감하게 볼 수 있으므로 8바이트 정렬도 함께 신경 써야 한다.

## 잘 된 점

헷갈리는 부분을 바로 코드로 밀어붙이기보다, 실행 흐름과 데이터의 모양을 먼저 그림으로 잡으려 한 점이 좋았다. `process_exec()`의 `f_name`이 어디서 오는지, `load()`의 `file_name`이 무엇을 받는지, `load()`의 `success`가 어떻게 `process_exec()`로 돌아오는지처럼 함수 사이 값 전달을 차근차근 확인했다.

또 테스트 실패를 한 덩어리로 보지 않고 단계별로 나눠서 볼 수 있게 되었다.

- 컴파일이 되는지 확인한다.
- 디버그 출력으로 파싱 결과를 확인한다.
- 유저 스택 구성과 레지스터 전달을 확인한다.
- `process_wait()` 때문에 timeout이 나는지 구분한다.
- 정식 테스트 전에는 불필요한 `printf`를 제거한다.

이렇게 나누면 `args-*` 테스트가 실패했을 때도 어디부터 의심해야 하는지 훨씬 명확해진다.

## 아쉬운 점

아직 구현 관점에서는 완성된 체크리스트가 더 필요하다. 특히 커맨드라인 원본을 `strtok_r`로 수정하는 순간 이후에 어떤 버퍼를 파일 이름용으로 쓰고, 어떤 버퍼를 인자 배열 구성용으로 쓸지 명확히 정해야 한다.

또 PintOS 테스트는 출력 비교가 엄격하기 때문에 디버그 출력 관리가 중요하다. 중간 확인용 `printf`는 학습에는 도움이 되지만, 테스트 통과를 확인하기 전에는 반드시 제거해야 한다.

`args-*` 테스트를 끝까지 확인하려면 argument passing만이 아니라 `process_wait()` 흐름도 함께 봐야 한다. 스택 구성이 맞더라도 wait 쪽이 무한루프이거나 종료 상태 처리가 안 되어 있으면 timeout으로 보일 수 있기 때문이다.

## 이번 주 한 줄 회고

이번 주의 핵심은 "문자열을 자르는 것"에서 멈추지 않고, 그 문자열이 유저 프로그램의 `argc`, `argv`로 도착하기까지의 전체 경로를 연결해서 이해한 것이다.
